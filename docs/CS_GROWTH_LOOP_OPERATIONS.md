# 💬 CS팀 + Growth Loop — 담당 구간 업무 상세

> **담당 Phase**: 지급/사후(Phase 7 일부) + Growth Loop(Phase 8)  
> **핵심 목표**: 지급 완료 후 고객 만족 → 소개/재유입 → 성장 루프 가동  
> **팀 구성**: CS팀원(사후관리·소개·VOC), 관리자(KPI 추적)

---

## 📊 Growth Loop 전체 구조

CS팀은 3가지 여정 전체에서 **지급 완료 후 성장 루프**를 담당합니다.

```
┌─────────────────────────────────────────────────────────────────┐
│               성장 루프 순환 구조 (Growth Loop)                    │
├─────────────────────────────────────────────────────────────────┤
│                                                                   │
│  3년환급 여정                소개 여정              간편청구 여정     │
│  ──────────────            ──────────            ──────────────   │
│                                                                   │
│  S15 사후관리                R12 사후관리           Q7 지급추적      │
│    ↓                          ↓                      ↓            │
│  S16 소개푸시 ─┐           R13 소개생성 ─┐        Q8 보장공백탐지  │
│    (신규유입)  │             (재소개)     │        │ (여정전환가능) │
│    ↓          │               ↓          │        │              │
│  S17 재유입   │            R14 성장루프  │        Q9 리텐션/소개  │
│  ──────────────┘            ──────────────┘        ──────────────  │
│       ↑                           ↑                    ↑           │
│       └───────────────┬──────────┬────────────────────┘           │
│                       ↓                                           │
│            신규 고객 유입 (R1, S1, Q1)                            │
│            ↓                                                      │
│         [루프 재개]                                              │
│                                                                   │
└─────────────────────────────────────────────────────────────────┘
```

## 🔄 S 여정(3년환급) — S15~S17 Growth Loop

### S15. 사후관리/정산

**담당**: CS팀 | **구현 상태**: ✅ 부분 구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 사후관리 대상 리스트 관리 | CS팀 | ✅ `Aftercare.tsx` | 지급 완료 건 중 콜 대상 필터링 (담당영업사원별) |
| [OFF] | 사후관리 전화 (만족도 조사) | CS팀 | 오프라인(전화) | 1~3분 통화: "지급되셨나요?", 만족도 5점 스케일 |
| [ON] | 고객 만족도 기록 | CS팀 | ✅ `Aftercare.tsx` | 콜 완료 후 만족도 점수 입력 (1~5) |
| [ON] | 수수료 정산 처리 | 관리자 | ⚠️ 부분 | 영업사원별 수수료 집계 (의존: S16 소개 건수) |

#### 🎯 실행 프로세스

1. **대상 선별** (자동)
   - 조건: `payment_date + N일` 이내 (기본값 7~14일)
   - 필터: 추적 가능한 전화번호 있는 고객만
   - 페이지: `Aftercare.tsx` → "사후관리" 탭 → 전화 가능한 건만 표시

2. **콜 스크립트**
   ```
   "○○고객님, 안녕하세요. 더바다 보험입니다.
   지난주 청구하신 금액이 계좌로 입금되셨나요?
   (응답 대기)
   혹시 통장에 안 보이시면 처리 중일 수도 있습니다.
   
   입금 확인하셨다면, 서비스에 얼마나 만족하셨는지 여쭤봐도 될까요?
   (1: 매우불만족 ~ 5: 매우만족)"
   ```

3. **만족도 기록**
   - 점수 입력 후 저장 → 즉시 KPI 반영
   - 만족도 4점 이상: 소개 대상 상향(S16 후보 우선)
   - 만족도 2점 이하: VOC 자동 생성 (CS → 고객관리)

4. **정산 처리** (관리자)
   - 월말 집계: 영업사원별 사후관리 참여도 + 소개 성공건수
   - 수수료 = 기본료 + 소개 인센티브

### S16. 소개 푸시 Trigger

**담당**: CS팀 + 시스템 | **구현 상태**: ⚠️ 부분 구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 소개 대상 고객 선별 | CS팀 | ⚠️ 수동 | 만족도 4점 이상 + 지급 후 N일 이내 |
| [EXT] | 소개 푸시 알림 발송 (앱/카카오) | 시스템 | ❌ 미구현 | 자동 Trigger 없음 — 수동 발송 필요 |
| [ON] | 소개 전환율 추적 | 관리자 | ⚠️ 1개만 | KPI: 소개율(발송 대비 R1 유입 비율) |

#### 🎯 실행 프로세스

1. **선별 로직**
   ```
   대상 조건:
   - 만족도 >= 4점 (S15에서 기록)
   - 지급완료일 + 7~14일 이내 (설정 가능)
   - 휴면 고객 제외 (최근 1년 내 거래 있는 건만)
   - 추천코드 미발급 (중복 발송 방지)
   ```

2. **소개 푸시 콘텐츠** (예시)
   ```
   [제목] "○○님, 더바다 보험으로 추가 환급받으세요!"
   
   [본문] "지난 청구가 도움이 되셨다면,
   가족이나 지인분도 같은 서비스로 환급받을 수 있습니다.
   
   추천 링크로 초대하고 추천보상도 받으세요!
   
   [버튼] '추천링크 보기' → R1(소개 여정) 시작"
   ```

3. **발송 채널**
   - **앱 푸시**: 앱 설치한 고객 우선
   - **카카오톡 채널**: 앱 미설치 고객
   - **SMS**: 폴백 (카카오 미동의 시)

4. **전환율 추적** (KPI)
   ```
   KPI: 소개 전환율 = (R1 신규 고객) / (S16 푸시 발송 수) × 100%
   
   목표: 월 15~20% 이상
   대시보드: 관리자 → KPI → "소개 전환율" 트래킹
   ```

### S17. 재유입

**담당**: 시스템 | **구현 상태**: ❌ 미구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 재유입 고객 식별 | 시스템 | ❌ 미구현 | 추천코드로 R1 → S 여정 매칭 |
| [ON] | 재유입→S4 자동 배정 | 시스템 | ❌ 미구현 | 추천인의 담당영업사원에게 자동 배정 |

#### 🎯 실행 프로세스

1. **재유입 감지**
   ```
   감지 로직:
   - R1(소개 여정) 신규 고객 발생
   - 추천코드 확인 → 기존 S 고객 매칭
   - 이전 여정(S1~S17) 기록 확인
   ```

2. **여정 연결** (S4로 단계 건너뜀)
   ```
   기존 흐름: S1 → S2 → S3 → [S4 선별] → S5 TM
   
   재유입: R1 추천 → S4(선별 스킵) → S5(TM)
   (= 추천인 담당자가 그대로 담당, 선별과정 생략)
   ```

3. **담당자 배정**
   - 추천인의 기존 담당영업사원에게 자동 배정
   - 관리자 개입 불필요 (자동)

## 🔄 R 여정(소개) — R12~R14 Growth Loop

### R12. 사후관리 및 정산

**담당**: CS팀 | **구현 상태**: ❌ 미구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 소개 고객 사후관리 | CS팀 | ❌ 미구현 | S15와 동일 (만족도 조사) |
| [ON] | 소개 수수료 정산 | 관리자 | ❌ 미구현 | 추천인 + 피추천인 양쪽 수수료 계산 |

#### 🎯 실행 프로세스

1. **사후관리** (S15 동일)
   - R11(청구서/지급) 완료 후 7~14일 내 콜
   - 만족도 기록
   - 다만 **추천인도 통보** (소개 성공 확인)

2. **수수료 구조** (소개 여정 특수)
   ```
   추천인 수수료 = 피추천인 청구액의 3~5% (설정값)
   + 추천인이 다시 추천했을 경우 추가 인센티브
   
   정산 대상:
   - R1 → R11(지급) 완료한 건만 (재무 임팩트 없는 추적 제외)
   - 월말 정산
   ```

### R13. 소개 생성/가족 연동

**담당**: CS팀 | **구현 상태**: ❌ 미구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 2차 소개 추적 (피추천인→다시 소개) | CS팀 | ❌ 미구현 | R12 사후관리 시 "다시 소개 의향" 확인 |
| [ON] | 가족 연동 제안 (배우자/형제 등) | CS팀 | ❌ 미구현 | 배우자, 형제자매 등 추가 발굴 |
| [ON] | 48시간 내 요청 완료율 추적 | 관리자 | ❌ 미구현 | KPI: 요청→추천코드 발급 시간 |

#### 🎯 실행 프로세스

1. **2차 소개 발굴** (R12 사후관리 중)
   ```
   콜 스크립트 추가:
   "○○님, 혹시 가족이나 지인 중에
   보험 청구할 일 있으신 분이 계세요?
   
   (응답에 따라)
   "네, 있으신가요? 그분도 더바다로
   청구해드릴 수 있습니다.
   함께 연결해드릴까요?"
   
   → 응답 Yes → R13 단계 진행 (2차 소개)"
   ```

2. **가족 연동 제안** (별도 플로우)
   ```
   대상:
   - 배우자: 동일 가구 → 함께 청구 가능
   - 형제자매: 별도 처리, 소개 링크 공유
   - 자녀(성인): 독립적 R1로 처리
   
   제안 시점: R12 사후관리 중 또는 R13 소개 후
   
   가족 연동 효과:
   - LTV 증가 (한 고객군 다중 청구)
   - 소개율 증가 (가족 네트워크 활용)
   ```

3. **완료율 KPI**
   ```
   KPI: 48시간 내 추천코드 발급률
   
   측정:
   - 고객이 "소개 의향" 표시
   - CS팀이 48시간 내 추천코드 생성
   - 비율 = (48시간 내 완료) / (전체 의향) × 100%
   
   목표: 95% 이상
   ```

### R14. 소개 재유입/성장 루프

**담당**: 관리자 + 시스템 | **구현 상태**: ❌ 미구현

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | CAC 절감 효과 / LTV 기여도 추적 | 관리자 | ❌ 미구현 | R1→R11 고객의 수익성 분석 |
| [ON] | 재유입→R1 루프 연결 | 시스템 | ❌ 미구현 | R13 2차 소개 → R1 신규 고객 매칭 |

#### 🎯 실행 프로세스

1. **CAC/LTV 추적** (분석용)
   ```
   지표 1: CAC(고객획득비용) 절감
   - 기존 광고/마케팅 대비 소개 고객 CAC
   - 예: 광고 CAC 50,000원 vs 소개 CAC 5,000원 (90% 절감)
   
   지표 2: LTV(고객생애가치) 증대
   - R1 고객 → R11 청구액
   - 추천인 수익(수수료) 포함
   - 추적 기간: 최소 6개월
   
   대시보드: 관리자 → KPI → "소개 여정 수익성" 섹션
   ```

2. **루프 폐쇄** (R13 → R1)
   ```
   재유입 시나리오:
   
   R13에서 "배우자 소개 의향" → 추천코드 발급
   → 배우자가 앱 설치 → 추천코드 입력
   → R1(소개 여정) 신규 고객 발생
   → R2~R11 청구 프로세스 진행
   → R12 사후관리 (다시 R13으로 계속)
   
   이것이 "성장 루프" = 자가증식 고객 획득
   ```

3. **루프 가속화** (지표)
   ```
   루프 속도 = (R13 소개 의향) / (R12 사후관리 고객)
   
   루프 강도 = (R1 신규 고객) / (R13 의향) 
           × (R11 지급액 / R12 사후관리 고객)
   
   목표: 분기별 루프 강도 10% 증가
   ```

## 🔄 Q 여정(간편청구) — Q8~Q9 Growth Loop

### Q8. 보장 공백 탐지

**담당**: 시스템 + CS팀 | **구현 상태**: ❌ 미구현 (개념 자체 없음)

#### 📋 업무 목록

| 구분 | 업무 | 담당 | 어드민 | 설명 |
|------|------|------|--------|------|
| [ON] | 보장 공백 분석 (현재 보험 vs 필요 보장) | 시스템 | ❌ 미구현 | 심평원 청구 기록 vs 보험 증권 비교 |
| [ON] | 보장 공백 발견 시→3년환급 서비스 연계 | 시스템 | ❌ 미구현 | 여정 전환: Q9 → S1(3년환급 유입) |
| [ON] | 보장 공백 발견율 / 연계율 KPI | 관리자 | ❌ 미구현 | KPI: 발견율 %, 연계율 % |

#### 🎯 실행 프로세스

1. **보장 공백 정의**
   ```
   보장 공백 = 청구 가능한데 보험에 없거나 제한된 항목
   
   예시:
   - 고객: 일반 상해보험만 가입 (뇌/심장 질환 제외)
   - 심평원 기록: 뇌졸중 진료비 40만원 발생
   - 분석: "뇌병변 특약 없음 → 앞으로 비용 부담할 위험 HIGH"
   - 권고: "3년환급+뇌질환특약" 추가 청구 제안
   ```

2. **탐지 로직**
   ```
   입력:
   - Q7(지급 추적) 완료 시 고객의 최종 청구 항목
   - 고객의 현재 가입 증권 정보
   
   분석:
   - 청구된 항목 중 보험 가입 범위 확인
   - 청구 불가능한 항목 자동 식별
   - 공백 위험도 스코어링 (높음/중간/낮음)
   
   출력:
   - "보장 공백 발견: O/X"
   - "권장 추가 서비스: 3년환급 (특약: 뇌질환, 암진단비 등)"
   ```

3. **여정 전환** (Q9 → S1)
   ```
   현재: Q9(리텐션) → 고객 유지만 고려
   
   개선: Q9 → [보장 공백 발견?]
         ├─ Yes → S1(3년환급) 자동 생성
         │         (추천 특약 정보 포함)
         └─ No → Q9 리텐션 계속
   
   구현: JourneyContext에서
   transitionToJourney('refund', {...})
   호출 → S1 스텝 자동 배정
   ```

4. **발견율/연계율 KPI**
   ```
   KPI 1: 보장 공백 발견율
   = (Q8에서 공백 발견) / (Q7 완료 건) × 100%
   목표: 20~30%
   
   KPI 2: 여정 연계율
   = (S1 전환된 건) / (공백 발견 건) × 100%
   목표: 50% 이상 (="고객이 추가 청구에 동의한 비율")
   ```

### VOC 처리 프로세스

```
VOC 발생
  ↓
[1] CS팀 초수 대응 (전화/메시지)
  ├─ 해결 가능 (지급 확인, 안내 재전달)
  │  └─ 종료
  └─ 해결 불가 (버그, 규정 이슈)
     ↓
[2] 관리자 검토
  ├─ 일시적 이슈 → 메모 기록
  └─ 반복/심각 → 버그 리포트 또는 정책 변경
```

### VOC 통계 및 개선

```
월별 VOC 집계:
- 유형별 발생 수
- 해결 시간 (초수 대응 시간)
- 재발 비율

개선 방향:
- 발생 많은 부분 자동화 (예: 지급 상태 조회 API)
- 소개 프로세스 단순화
- 콜 스크립트 최적화
```
